Not a feature list — a mental model. How a sealed box runs your ASP.NET app anywhere.

Follow the flow

The one-sentence version

A container is a tiny computer for your app

Docker does not replace dotnet. It packages the app with a Linux OS and the ASP.NET runtime so “works on my machine” becomes “works in this box.”

Without Docker

Your laptop installs the SDK, restores NuGet, and runs the process.

  • Depends on your OS & installed runtimes
  • dotnet run on the host
  • Different every developer machine
vs

With Docker

A recipe builds a sealed environment; you start that environment.

  • Same Linux + ASP.NET inside the box
  • dotnet YourApp.dll in the container
  • Identical on laptop, CI, and server

The flow

Four words to remember

Image is the recipe. Container is the running meal. Dockerfile writes the recipe. Compose starts the meal on your desk.

1

Dockerfile

Instructions: copy code, run dotnet, set entrypoint

2

Image

Frozen snapshot — layers of files ready to start

3

Container

That snapshot running as a process with its own filesystem

4

Port

Map host 8080 → container 8080 so you can curl it

Mapped to .NET

You already know these commands

A Dockerfile for ASP.NET is mostly the same steps you run locally — just inside someone else’s clean machine image from Microsoft.

01

Borrow a machine with the SDK

FROM mcr.microsoft.com/dotnet/sdk means: start from Microsoft’s image that already has the .NET SDK. No installing Visual Studio on a server.

≈ having the SDK installed
02

Restore, then publish

Inside that image you run dotnet restore and dotnet publish — exactly like CI. The output is your DLL and content files, not the whole SDK.

dotnet restore → dotnet publish -c Release
03

Switch to a smaller machine to run

Multi-stage builds throw away the SDK image and copy only the publish folder into mcr.microsoft.com/dotnet/aspnet. Think: build kitchen vs serve dining room.

SDK image builds · aspnet image runs
04

Start the process

ENTRYPOINT ["dotnet", "YourApp.dll"] is dotnet run without the SDK — just the runtime host and your published output. Listen on a port (often 8080).

dotnet YourApp.dll

Dockyard is that pattern, named

Same mental model. The files in this repository are one concrete ASP.NET API following the flow above.

SDK restore + publish → Dockerfile stages restore / build
Slim ASP.NET runtime → Dockerfile stage runtime
How you start it locally → compose.yaml → port 8080
Proof it works → GET /api/v1/runtime

Feel it

Build the box, then knock on the door

Compose reads the Dockerfile, builds the image, starts a container, and publishes the port to your machine. 

$ docker compose up --build

$ curl http://localhost:8080/api/v1/runtime
